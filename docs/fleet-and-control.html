cc-fleet-miner · protocol & internals

How fleet talks to control

One control computer and a handful of mining turtles run two different programs that never call each other — they only trade typed messages over rednet. This is the companion to the operator guide: what those messages are, who says what, and how a run actually unfolds.

Protocol: ccfleet (rednet) Roles: 1 hub · 2–5 workers GPS: not required Companion: control.html

control control.lua

The hub. Splits the box into strips, hands them out, tracks progress, reassigns dead work. Never digs. One per fleet.

fleet fleet.lua

A worker. Announces itself, waits for a strip, mines it relative to where it stands, reports in, obeys stop/pause. One per turtle.

1Two programs, one conversation

It's a hub-and-spoke system. You run control on one computer with a modem, and fleet on each turtle. Neither program calls the other or shares any memory — they communicate only by broadcasting and sending small message tables (called envelopes) over a single named rednet protocol, "ccfleet".

That hard separation is the whole point of the design: because the turtle-facing logic (the worker) and the hub logic (the coordinator) both talk to an injected messaging layer instead of to rednet directly, the entire fleet can be tested against a fake in-memory bus, off Minecraft. What you're reading here is that message contract.

The division of labour

control decides who mines what and watches for trouble. fleet decides nothing about the fleet — it only knows its own strip and its own position. All coordination lives in the hub; a turtle is deliberately dumb.

2The shared coordinate frame (why fleet takes x y z)

This is the one idea to hold in your head, because it explains the turtle's launch arguments. The hub splits your box by X into contiguous, non-overlapping column-strips — one slab per turtle:

WIDTH → (+X, to the turtles' right)  ·  a 6×L box, 2 turtles
STRIP s1 · width 3
Aturtle A
origin x=0, y=1, z=0
> fleet 0 1 0
STRIP s2 · width 3
Bturtle B
origin x=3, y=1, z=0
> fleet 3 1 0
↓   DEPTH goes straight down (−Y)   ↓

Here's the subtle part: a turtle mines relative to wherever it is physically standing. The quarry pattern never reads an absolute coordinate — it just clears width × length × depth starting from the turtle's own body. So the x y z you type into fleet is used for exactly one thing: it's the turtle's claimed position in the hub's frame, so the coordinator can position-match — give the strip whose origin equals your pose to the turtle standing there.

Type the right coords and turtle B (at x=3) is handed the x=3 strip. This is also the project's structural collision avoidance: turtles never path around each other because they physically can't overlap — they were pre-placed in separate X-strips. There's no runtime dodging to go wrong.

Get the coords wrong…

…and a turtle can be matched to the wrong-width strip and overlap its neighbour, or sit idle because no strip origin matches its pose. The coords are the hub's only way to know which physical turtle should get which slab.

3The six messages

Every packet is an envelope — a plain table with a fixed shape. The payload differs per type; everything else is routing and versioning:

{
  v       = 1,            -- protocol version
  type    = "ASSIGN",     -- one of the six types below
  from    = 4,            -- sender's computer id
  to      = 7,            -- recipient id, or nil for a broadcast
  seq     = 12,           -- per-sender counter
  payload = { job = {...} } -- type-specific contents
}

Six types flow between the two programs. Direction and delivery matter — a broadcast reaches everyone on the protocol; a directed send goes to one id:

MessageFrom → ToDeliveryPayloadMeaning
REGISTER turtle→all broadcast {pos, label} "I exist, I'm here, call me Dozer."
ASSIGN control→turtle directed {job} "Mine this strip" (width / length / depth).
PROGRESS turtle→control directed {pos, fuel, mined, job} "Still alive, here's my status." Also the heartbeat.
DONE turtle→control directed {job} "My strip is clear."
ERROR turtle→control directed {reason, job} "I failed" (out of fuel, stuck, hit bedrock).
CONTROL control→turtle(s) directed / broadcast {cmd} pause · resume · stop · return.
Two consequences worth internalising

REGISTER is a broadcast because a fresh turtle doesn't yet know the hub's id. It learns that id from the first ASSIGN or CONTROL it receives, then sends all its PROGRESS / DONE / ERROR straight there.

Which means only the control computer ever receives progress — those are point-to-point sends, not broadcasts. A second computer eavesdropping on the protocol would see the REGISTER beacons but none of the status traffic.

4Inside control: the pump

The station runs three cooperative loops at once (via parallel.waitForAny): one drives the coordinator, one reads your keystrokes, one redraws the dashboard. The engine is the first one — it just calls coord:step() over and over until the run is complete. Each step() is one full pump:

while not coord:isComplete() do coord:step(1) end
  1. Drain the inbox. Read every waiting message (the first receive may block up to 1s so the loop paces itself) and update the roster + per-turtle status from each.
  2. Liveness sweep. Any mining/assigned turtle silent longer than staleAfter is marked stale, and its strip is requeued for reassignment.
  3. Assign. Hand every queued strip to a free turtle, preferring one whose pose matches the strip's origin.
  4. Dispatch when the expected roster has gathered: partition the current box into strips and queue them.
  5. Advance. When every strip of the box is terminal, pop the next queued box — or mark the whole run complete.
  6. Flush. If anything durable changed, save the job state to disk (so a reboot resumes).

The other two loops are thin: keystrokes P/R/S/H call pauseAll / resumeAll / stopAll / returnAll (each broadcasts a CONTROL command and flips a flag), and the render loop paints the table every 0.3s.

5Inside fleet: the worker loop

A turtle's life is much simpler. While it has no work it loops on a short cycle; once it's mining it's busy inside a single blocking call and reports out through a hook.

While idle

tick(): register() then step()
  1. Broadcast REGISTER with its pose + label.
  2. Wait up to heartbeat seconds for one message. An ASSIGN wakes it instantly; a timeout just loops.
  3. Because register repeats each cycle, it doubles as a heartbeat — a dropped or too-early registration heals itself, and a rebooted turtle rejoins on its own.

On ASSIGN → while mining

_onAssign(job): run the quarry
  1. Send an immediate PROGRESS (0 mined) so the dashboard flips to mining at once.
  2. Run the quarry — this call blocks until the strip is done.
  3. Every progressEvery blocks a hook fires: send PROGRESS, then poll for a CONTROL command.
  4. On finish → final PROGRESS + DONE. On failure → ERROR.
Why the hook exists

Mining is one long blocking call, so the only moment a busy turtle can notice a pause or stop is inside that throttled hook. That's why control response isn't instant — it lands within about progressEvery blocks (default 8). On a PAUSE the turtle parks right there and blocks until RESUME or STOP arrives.

6A full run, message by message

Here is the whole conversation for a 6-wide box and two turtles — hub actions sit on the centre line, messages cross between the lanes, coloured by who's speaking.

control.lua  (hub)
fleet.lua  (turtles A, B)
① Gather
REGISTER ▸ broadcast
A · pos (0,1,0) · "A"
REGISTER ▸ broadcast
B · pos (3,1,0) · "B"
roster = {A, B} ≥ expect (2)
② Dispatch
split box → s1 (x0, w3), s2 (x3, w3)
pose-match: A ↔ s1, B ↔ s2
ASSIGN ▸ A
job s1 · w3 × L × D
ASSIGN ▸ B
job s2 · w3 × L × D
③ Mine
PROGRESS ▸ control
A · mined 0 → flips to "mining"
PROGRESS ▸ control
B · mined 0 → "mining"
PROGRESS × N
A · mined 8, 16, 24 … fuel, pos
PROGRESS × N
B · mined 8, 16, 24 …
④ Finish
DONE ▸ control
A · job s1 clear
DONE ▸ control
B · job s2 clear
both strips terminal → phase = complete → station exits

If you'd pressed P partway through phase ③, the hub would have broadcast a CONTROL pause; each turtle notices it at its next PROGRESS hook and parks mid-strip until a CONTROL resume arrives.

7When things go wrong

The interplay earns its keep exactly here — the message loop is also the failure-recovery loop.

A turtle goes silent

Chunk unload, crash, or a stall with no ERROR. Its PROGRESS pings stop; after staleAfter seconds the hub marks it stale and requeues its strip. A free turtle — ideally a spare pre-placed at that strip's corner so the pose still matches — is re-ASSIGNed, up to maxAttempts before the strip is given up as failed so the run can't hang forever.

You steer the fleet

P/R/S/H broadcast a CONTROL command. Pause parks turtles mid-strip (reversible); Stop abandons in place; Home returns each turtle to its start pose first. All land within ~progressEvery blocks, since that's when a mining turtle next checks its inbox.

The hub reboots

Job state is on disk. restore() rehydrates it and seeds a status entry per in-flight strip. When turtles re-REGISTER (their heartbeat), any still holding an in-flight job is re-ASSIGNed the same job id — which the worker recognises as a duplicate and ignores, so nothing double-mines.

Idempotent by id

Strip ids are deterministic (box-s1, box-s2, …), so re-sending an ASSIGN is always safe. The worker keeps a seen set; a re-ASSIGN of a finished strip just re-acks its DONE. This is what makes reboots and lossy rednet survivable.

8The one CC gotcha that shaped both loops

ComputerCraft's os.sleep — and any filtered os.pullEvent — silently discards rednet_message events that arrive while it's waiting. A naïve "receive → render → sleep" loop would drop turtle messages and the dashboard would freeze. That single fact dictates the concurrency on both ends:

  • control uses parallel.waitForAny — parallel hands each event to every waiting coroutine, so the sleeping render loop can't starve the comms loop's receive.
  • fleet paces its idle loop on the blocking receive itself (the heartbeat timeout), never on os.sleep.
If you extend either program

Never drop an os.sleep into a loop that also needs to receive rednet. Pace on the receive's timeout, or add another coroutine under parallel. This is the bug that froze the dashboard the first time around.
