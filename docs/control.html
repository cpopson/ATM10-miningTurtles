1What control.lua is

control.lua is the brain of the fleet. It runs on a single computer with a modem and does four things at once:

Plans the job

An on-screen wizard collects the pattern, box dimensions, origin, and turtle count — then splits the box into non-overlapping column-strips, one per turtle.

Assigns & tracks

As turtles register over rednet, it hands each one a strip, watches their progress heartbeats, and re-assigns any strip that goes silent.

Shows a dashboard

A live, flicker-free table — one row per turtle (label, state, fuel, blocks mined, current job) plus queue and failure counts.

Steers the fleet

Single keystrokes pause, resume, stop, or send the whole fleet home. Job state is saved to disk every pump, so a reboot resumes automatically.

Two halves of one system

control.lua is the coordinator. Each turtle runs fleet.lua, the worker. This guide covers the whole loop: you run control on the computer and fleet on every turtle.

2What you need

MachineRequirements
1× Control computer A regular (or advanced) computer with a wireless or ender modem attached to any side. A monitor is optional — the dashboard renders on the computer's own terminal.
2–5× Mining turtles Each needs a modem (same kind — wireless or ender), fuel (coal/charcoal → refuel all), and ideally a chunk loader so it keeps running while you're away. An Advanced Mining Turtle gives colored output.
Ender Chests (optional) One in turtle slot 16 auto-dumps loot in place when the inventory fills. Pair it with a matching chest at base for hands-free storage.
GPS not required

Turtles track their position from a start pose you supply — no GPS towers needed. That's why you pass each turtle its coordinates when you launch it (see step 7).

3Get the code in-game

Every machine — the control computer and each turtle — needs the project files. Bootstrap once with update.lua, then re-run update after any change.

  1. Make sure BASE in update.lua points at your repo's raw-content URL (ending in /).
  2. On a fresh computer or turtle, pull the updater once: 
    > wget https://raw.githubusercontent.com/cpopson/ATM10-miningTurtles/master/update.lua update.lua
> update
  3. After every push to GitHub, just run update again on each machine.

CC:Tweaked has HTTP enabled and whitelists GitHub by default, so wget works out of the box.

4Understand the geometry (read this first)

Getting the box direction right in your head before you place turtles saves a lot of re-digging. Everything is relative to how a turtle faces:

DimensionDirectionAxis
Widthto the turtle's right+X
Lengthbehind the turtle+Z
Depthdown−Y
 
            WIDTH  (+X, turtles' right)  →
        ┌───────┬───────┬───────┐   ▲ turtle here, facing "into" the page
        │ strip │ strip │ strip │   │  (one block ABOVE the top corner)
LENGTH  │   0   │   1   │   2   │   │
(+Z,    │  T#1  │  T#2  │  T#3  │   each turtle clears its own column-strip
behind) │       │       │       │   top-to-bottom — columns never overlap.
        └───────┴───────┴───────┘
                                 DEPTH goes straight DOWN (−Y)

The coordinator splits the box into column-strips by X range and gives each turtle one slab to clear top-to-bottom. Because the strips never share a column, collision avoidance is structural — there's no runtime turtle-to-turtle dodging to go wrong.

Heading convention

0 = north (−Z), 1 = east (+X), 2 = south (+Z), 3 = west (−X). Coordinates match Minecraft/GPS axes. Turtles launch facing heading 0 by default.

5Start the control station

On the control computer, just run:

> control

On launch it does one of two things:

  • No saved job → it drops straight into the setup wizard (step 6).
  • A running job was saved (the computer crashed or rebooted mid-run) → it asks: 
    Saved job found. [R]esume or [N]ew?
    Press R (or Enter) to pick up exactly where it left off, or N to discard it and set up a fresh job.
No modem?

If you see "No modem found", attach a wireless or ender modem to any side of the computer and run control again. The station scans all six sides automatically.

6The setup wizard

The wizard asks a few questions, validates each answer, and re-asks anything invalid. It asks for the turtle count once, then walks one box at a time — you can queue several boxes.

PromptWhat to enter
Turtles in the fleetHow many turtles you'll launch (2–5). This is the number the station waits for before assigning work.
Patternquarry — clears the whole box. (Branch & tunnel patterns are on the roadmap.)
Width / Length / DepthThe box size in blocks — see the geometry section. Whole numbers ≥ 1.

Multi-box caveat

Each turtle mines relative to its own position, so a queue of boxes tiles cleanly only when the boxes share an origin region (re-runs, stacked layers). Boxes at different physical locations need the turtles repositioned and restarted between them — full relocation waits on GPS. For your first job, queue just one box.

7Place & start the turtles

With the station waiting, set up each turtle. For turtle i:

  1. Place it one block above the top corner of its strip, facing the box (heading 0 / north by default).
  2. Fuel it: coal/charcoal in a slot, then refuel all. It needs roughly width × length × depth for its slab plus the return trip.
  3. (Optional) Drop an Ender Chest into slot 16 for auto-dump.
  4. (Optional but recommended) Give it a name so the dashboard shows it: label set alpha.
  5. Launch the worker with that corner's coordinates: 
    > fleet <x> <y> <z>
-- e.g. the turtle sitting at the strip corner (2,1,0):
> fleet 2 1 0
Turtle at (2,1,0). Registering with control...

The <x> <y> <z> are the turtle's position in the same coordinate frame the control computer uses. The coordinator position-matches each turtle to the correct strip, so you don't assign strips by hand — you just tell each turtle where it is. Omit the args and it defaults to (0,0,0).

The handshake

Each turtle broadcasts REGISTER → the station replies with ASSIGN (its strip) → the turtle sends PROGRESS every few blocks (this doubles as its heartbeat) → DONE when its strip is clear, or ERROR if it gets stuck. A turtle that's still waiting re-announces itself every ~10s, so an early or dropped REGISTER heals itself.

9Keyboard controls

The dashboard responds to single keystrokes at any time — they act on the whole fleet at once:

KeyAction
PPause — turtles park mid-strip and hold. Reversible.
RResume — paused turtles pick their strips back up.
SStop — terminal. Turtles abort exactly where they are.
HHome — every turtle returns to its start block, then stops.
QQuit the station. Turtles keep running their last command.
Stop latency

Turtles check for control commands between PROGRESS pings, so a Pause/Stop lands within about progressEvery blocks (default 8) — not instantly, but quickly.

10Pause, stop & resume

Two independent safety nets keep a long run recoverable:

Reboot-safe station

The coordinator writes its job state to disk every pump. If the control computer crashes or you reboot it, re-run control and press R at the "Saved job found" prompt — it resumes, and turtles re-sync from their next heartbeat.

Self-healing assignment

If a turtle goes silent past staleAfter seconds (default 30), the coordinator reassigns its strip to another turtle. After maxAttempts failures (default 3) a strip is marked failed so one hopeless slab can't stall the whole job.

When every strip is done or failed, the station prints Run complete. and exits. If you quit early with Q or stop with S, it prints Station stopped.

11Tuning (config.lua)

All the timing and slot knobs live in one file, config.lua. Change a value, run update on the turtles and the control computer, and it takes effect everywhere.

KeyDefaultWhat it controls
protocol"ccfleet"The rednet protocol name shared by the computer and every turtle. All machines must match.
heartbeat10Seconds between a waiting turtle's re-REGISTER announcements.
staleAfter30Seconds of silence before the coordinator reassigns a strip. Keep it well above heartbeat.
progressEvery8Blocks mined between PROGRESS pings — also the Stop/Pause response latency.
chestSlot16Inventory slot reserved for the Ender Chest used by auto-dump.
maxAttempts3Reassignment attempts before a strip is marked permanently failed.

12Troubleshooting

SymptomFix
"No modem found" on control or fleet Attach a wireless/ender modem to any side and re-run. Both ends need the same kind.
Station stuck at "Waiting for N turtles to register…" Confirm each turtle is actually running fleet, its modem is attached, and every machine shares the same protocol in config.lua. Wireless modems also have a range limit — ender modems are unlimited.
A turtle shows idle and never gets a strip Its reported position doesn't match any unassigned strip. Re-check the fleet x y z coords against the box origin and geometry.
Turtle stops early / ERROR Usually out of fuel or hit bedrock. Refuel (refuel all) — it needs ~width × length × depth plus the return trip — and make sure the box bottom is above bedrock.
Loot dropping on the ground Put an Ender Chest in slot 16 (or the chestSlot you configured) so full inventories dump in place instead of overflowing.
Turtles stop when you walk away They're unloaded. Use a chunk loader / chunky turtle to keep them running while you're gone.
